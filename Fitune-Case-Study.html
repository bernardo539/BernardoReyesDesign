Building Fitune's experience platform from scratch

Token architecture, a 95%-production-ready component system, governance, and an AI-assisted delivery pipeline — for a multi-product B2B SaaS.

01 / The Problem

No shared design language across two products

When I joined Fitune, the product had no shared design language. Engineering and design were working from inconsistent references, handoff cycles were long, and there was no path to support dark mode or white-label theming for different studio operators across two separate products — Admin and Webapp.

My mandate: build the experience platform from scratch — token architecture, component system, governance model, and a delivery pipeline that could scale without adding headcount.

02 / Architecture

A 4-layer token system for multi-mode, multi-tenant scale

A flat token system wouldn't support the product's requirements. I designed a 4-layer architecture where each layer has a single responsibility and a single set of consumers.

The 3-layer alternative would have required duplicating every token for each mode and each tenant. The 4-layer model separates concerns: changing the brand doesn't touch mode logic, and adding a tenant doesn't touch the semantic layer.

03 / Component System

95% production-ready at handoff

Every component that exits the pipeline arrives at engineering at roughly 95% completion. Engineering inserts the output into staging and promotes to production — no interpretation work, no separate implementation spike.

This works because the pipeline enforces strict output contracts. The remaining 5% is intentional: integration-specific data wiring and API bindings handled during staging — the pipeline owns visual and interaction completeness, not business logic.

04 / Feature Delivery

45% edge-case coverage, and growing

Beyond individual components, the pipeline supports full feature prototypes — complete flows with edge cases, empty states, error states, and loading states documented and ready for review. The token and component foundation had to be stable before feature-level delivery could accelerate. That foundation is now in place.

05 / Governance

Three mechanisms against fragmentation

A token system without governance drifts. I built three lightweight mechanisms to keep the system coherent as it grew.

External library policy

Dependencies earn adoption

Any dependency must meet 5 criteria before adoption — usage volume, disproportionate rebuild cost, library stability, single adoption unlocking multiple components, and mandatory wrapper encapsulation so any library can be swapped without touching consumers.

Component promotion

Components earn their place

Nothing enters the shared system without passing a gate: token audit, light/dark validation, accessibility review, and stale-import detection across the codebase.

08 / What's Next

AI-native interaction patterns

The system handles static and interactive UI well. The next platform investment is AI-native interaction patterns — streaming states, agent-to-human handoffs, uncertainty cues, and trust signals. As AI-powered features become a priority, these patterns need to be defined at the system level, not solved one-off per team.

These are the questions I'm building toward.
